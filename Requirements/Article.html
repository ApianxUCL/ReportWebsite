Initial Requirements

The Initial Project Briefing

The initial project briefing we received was as follows: Healthcare drone porter interface, simulating porters around a hospital with multiple building sites, and can make use of AirTags, BLE devices and other ways of beacon route finding. Design a front end UI that is cross platform, as well as a SharePoint solution with equivalent UI to do an A-B test for comparison of results. Mimicking the UCL buildings to begin with will be a good starting point. Both packages should enable a hospital to plot locations of porters, and issue requests to them via Twilio, demonstrate queue and prioritisation of work, and integrate with other communication services.

Requirements

Using the briefing, we tried to write a list of requirements to base a project on.

Functional requirements

Must have: User should be able to:
See the live locations of all the porters
See the delivery details (to be added) of each porter
Communicate with the porters (one-way) I.e. sending a collection request for samples coming in via drone
Nice to have: Communicate with porters (two-way) I.e. a porter to be able to decline a request
See the live locations of all crash calls/integrate into Crash 2222 calls

Non-functional requirements

Availability and fault tolerance with backups; since the functionality of the app can be a matter of life and death, it should have virtually zero downtime.
Extendibility; it should be easy to adapt the website to different buildings
Usability; the interface should be intuitive and easy to use
